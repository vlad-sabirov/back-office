## 1. Telegram-уведомления

### 1.1 Описание

Система Telegram-уведомлений для задач и событий календаря. Уведомления отправляются через существующий в проекте `TelegramService` (Telegraf). Включает мгновенные уведомления при действиях и крон-уведомления (напоминания, просрочка).

### 1.2 Уведомления по задачам

**Мгновенные (при действии):**

| Событие | Кому | Когда |
|---------|------|-------|
| Создание задачи | Исполнителю | Сразу |
| Изменение задачи (не самим исполнителем) | Исполнителю | Сразу |
| Смена статуса задачи | Автору (если сменил не он) | Сразу |
| Создание задачи (приоритет высокий/срочный) | Руководителю | Сразу (если `notifyBoss = true`) |
| Смена статуса задачи | Руководителю | Сразу (если `notifyBoss = true`) |

**Крон-уведомления задач:**

| Уведомление | Кому | Когда |
|-------------|------|-------|
| Напоминание за 3 дня до дедлайна | Исполнителю | Крон, ежедневно в 09:00 |
| Напоминание за 1 день до дедлайна | Исполнителю + руководителю (если `notifyBoss`) | Крон, ежедневно в 09:00 |
| Напоминание за 2 часа до дедлайна | Исполнителю + руководителю (если `notifyBoss`) | Крон, каждые 30 минут |
| Задача просрочена | Исполнителю + руководителю (если `notifyBoss`) | Крон, ежедневно в 09:00 |

### 1.2.1 Управление уведомлениями руководителя

Руководитель получает слишком много уведомлений, если подписан на все задачи всех подчинённых. Поэтому уведомления руководителю — **выборочные, per-task**.

**Поле `notifyBoss`** (Boolean) в таблице `crm_task`:
- Определяет, получает ли руководитель уведомления по этой конкретной задаче
- **По умолчанию:**
  - Приоритет `high` или `urgent` → `notifyBoss = true`
  - Приоритет `low` или `normal` → `notifyBoss = false`
- При смене приоритета — `notifyBoss` автоматически пересчитывается (но только если значение не было изменено вручную)

**Кто управляет:**

| Действие | Кто может |
|----------|-----------|
| Включить/выключить `notifyBoss` при создании задачи | Исполнитель (создаёт сам) или руководитель (создаёт подчинённому) |
| Изменить `notifyBoss` после создания | Исполнитель (чекбокс "Уведомлять руководителя") |
| Подписаться на задачу подчинённого | Руководитель (из списка задач подчинённого или из карточки задачи) |
| Отписаться от задачи | Руководитель |

**UI для руководителя:**
- В списке задач подчинённого: иконка 🔔/🔕 у каждой задачи — клик для быстрого вкл/выкл
- В карточке задачи: чекбокс "Получать уведомления" (видно только руководителю и автору)
- Фильтр "Только с уведомлениями" в списке задач

**UI для исполнителя:**
- При создании задачи: чекбокс "Уведомлять руководителя" (default по приоритету)
- В карточке задачи: чекбокс "Уведомлять руководителя"

**Модель данных — поля в crm_task:**

| Поле | Тип | Описание |
|------|-----|----------|
| notifyBoss | Boolean | Уведомлять руководителя (default: зависит от приоритета) |
| notifyBossManual | Boolean | Значение notifyBoss было изменено вручную (default: false) |
| reminderSent3Days | Boolean | Напоминание за 3 дня отправлено (default: false) |
| reminderSent1Day | Boolean | Напоминание за 1 день отправлено (default: false) |
| reminderSent2Hours | Boolean | Напоминание за 2 часа отправлено (default: false) |
| overdueNotified | Boolean | Уведомление о просрочке отправлено (default: false) |

**Формат Telegram-сообщений (задачи):**

```

"Подготовить КП для ООО Ромашка"

🏢 ООО "Ромашка"
```

## 2. Участники встреч (CalendarEventParticipant)

### 2.1 Описание

Возможность добавлять нескольких участников к встрече. Участники видят встречу в своих календарях, получают уведомления.

### 2.2 Модель данных — таблица: calendar_event_participant

| Поле | Тип | Описание |
|------|-----|----------|
| id | Int, PK | Автоинкремент |
| eventId | Int, FK → CalendarEvent | Событие |
| userId | Int, FK → User | Участник |
| status | String | "pending" / "accepted" / "declined" |
| createdAt | DateTime | Дата добавления |

### Индексы
- `(eventId, userId)` — уникальный, один пользователь = один раз на событие
- `(userId, status)` — поиск приглашений пользователя

### 2.3 Функциональность

- При создании встречи — поле "Участники" (MultiSelect сотрудников)
- Участники видят встречу в своём календаре с пометкой "Приглашение"
- Участник может принять / отклонить приглашение
- Автор видит статусы участников (кто принял, кто отклонил, кто не ответил)
- Assignee всегда является главным участником (не может отклонить)

### 2.4 Уведомления участников

| Событие | Кому | Когда |
|---------|------|-------|
| Приглашение на встречу | Участнику | Сразу |
| Участник принял/отклонил | Автору встречи | Сразу |
| Изменение времени/места встречи | Всем участникам | Сразу |
| Отмена встречи | Всем участникам | Сразу |
| Напоминание о встрече за 2 часа | Всем принявшим участникам | Крон |

### 2.5 API-эндпоинты

| Метод | URL | Назначение |
|-------|-----|-----------|
| POST | `/crm/calendar-event/:id/participants` | Добавить участников |
| DELETE | `/crm/calendar-event/:id/participants/:userId` | Удалить участника |
| PATCH | `/crm/calendar-event/:id/participants/respond` | Принять / отклонить приглашение |
| GET | `/crm/calendar-event/:id/participants` | Список участников события |
| GET | `/crm/calendar-event/invitations` | Мои приглашения (pending) |

### 2.6 Интерфейс

**В форме создания/редактирования встречи:**
- Новое поле: "Участники" — MultiSelect с поиском сотрудников
- Показывает аватарки выбранных участников

**В карточке встречи (просмотр):**
- Блок "Участники" — список с аватарками и статусами (✓ принял, ✗ отклонил, ⏳ ожидает)
- Для участника: кнопки "Принять" / "Отклонить"

**В календаре:**
- Встречи-приглашения отображаются полупрозрачными (пока не принято)
- После принятия — как обычная встреча
- Отклонённые — скрываются

**На дашборде:**
- Счётчик непринятых приглашений (бейдж)

## 3. Напоминания к событиям (CalendarReminder)

### 3.1 Описание

Пользователь может настроить индивидуальные напоминания для любого события — за 5 мин, 15 мин, 30 мин, 1 час, 1 день, или в произвольное время.

### 3.2 Модель данных — таблица: calendar_reminder

| Поле | Тип | Описание |
|------|-----|----------|
| id | Int, PK | Автоинкремент |
| eventId | Int, FK → CalendarEvent | Событие |
| userId | Int, FK → User | Кому напомнить |
| remindAt | DateTime | Когда напомнить |
| isSent | Boolean | Отправлено (default: false) |
| createdAt | DateTime | Дата создания |

### Индексы
- `(remindAt, isSent)` — крон-выборка неотправленных
- `(eventId)` — напоминания события
- `(userId)` — напоминания пользователя

### 3.3 Функциональность

- При создании/редактировании события — блок "Напоминания"
- Быстрые варианты: за 5 мин / 15 мин / 30 мин / 1 час / 1 день
- Можно добавить несколько напоминаний
- Произвольное время (DateTimePicker)
- Крон каждые 1 минуту проверяет `remindAt <= now() AND isSent = false`

### 3.4 API-эндпоинты

| Метод | URL | Назначение |
|-------|-----|-----------|
| POST | `/crm/calendar-event/:id/reminders` | Добавить напоминание |
| DELETE | `/crm/calendar-reminder/:id` | Удалить напоминание |
| GET | `/crm/calendar-event/:id/reminders` | Напоминания события |

### 3.5 Крон

`/cron/calendar-custom-reminders` — обработка пользовательских напоминаний. Выборка: `remindAt <= now() AND isSent = false`. Отправка через TelegramService, пометка `isSent = true`.

## 6. Повторяющиеся события (Recurrence)

### 6.1 Описание

Возможность создавать повторяющиеся события — ежедневные, еженедельные, ежемесячные встречи и звонки. Например: "Планёрка каждый понедельник в 10:00" или "Звонок клиенту каждый 1-й числа месяца".

### 6.2 Модель данных — изменения в calendar_event

| Поле | Тип | Описание |
|------|-----|----------|
| isRecurring | Boolean | Повторяющееся событие (default: false) |
| recurrenceRule | String? | Правило повторения (JSON) |
| recurrenceParentId | Int?, FK → CalendarEvent | Родительское событие (для экземпляров) |
| recurrenceExceptions | String? | JSON-массив исключённых дат |

### 6.3 Формат recurrenceRule (JSON)

```json
{
  "frequency": "daily" | "weekly" | "monthly" | "yearly",
  "interval": 1,
  "daysOfWeek": [1, 3, 5],
  "dayOfMonth": 15,
  "endType": "never" | "afterCount" | "untilDate",
  "endCount": 10,
  "endDate": "2026-12-31"
}
```

**Примеры:**
- Каждый день: `{ "frequency": "daily", "interval": 1, "endType": "never" }`
- Каждый Пн и Ср: `{ "frequency": "weekly", "interval": 1, "daysOfWeek": [1, 3], "endType": "never" }`
- Каждые 2 недели в Пт: `{ "frequency": "weekly", "interval": 2, "daysOfWeek": [5], "endType": "never" }`
- Каждый месяц 1-го числа, 10 раз: `{ "frequency": "monthly", "interval": 1, "dayOfMonth": 1, "endType": "afterCount", "endCount": 10 }`
- До конкретной даты: `{ "frequency": "weekly", "interval": 1, "daysOfWeek": [2], "endType": "untilDate", "endDate": "2026-06-30" }`

### 6.4 Логика работы

**Генерация экземпляров:**
- Повторяющееся событие хранится как одна запись (родитель) с `isRecurring = true`
- При запросе `/range` — сервер на лету генерирует виртуальные экземпляры по `recurrenceRule` для запрашиваемого диапазона дат
- Виртуальные экземпляры НЕ сохраняются в БД (экономия места)
- Экземпляр сохраняется в БД только при его индивидуальном редактировании (`recurrenceParentId` → родитель)

**Редактирование:**
- "Изменить это событие" — создаётся отдельная запись с `recurrenceParentId`, дата добавляется в `recurrenceExceptions` родителя
- "Изменить все события" — обновляется родительская запись
- "Изменить это и все последующие" — родитель обрезается `endDate`, создаётся новый повторяющийся родитель с изменениями

**Удаление:**
- "Удалить это событие" — дата добавляется в `recurrenceExceptions`
- "Удалить все события" — удаляется родитель + все экземпляры
- "Удалить это и все последующие" — родитель обрезается `endDate`

### 6.5 Интерфейс

**В форме создания/редактирования:**
- Чекбокс "Повторять"
- При активации: Select частоты (ежедневно / еженедельно / ежемесячно / ежегодно)
- Интервал (каждые N дней/недель/месяцев)
- Для еженедельно: MultiSelect дней недели (Пн — Вс)
- Для ежемесячно: число месяца
- Завершение: никогда / после N повторений / до даты

**При редактировании повторяющегося события:**
- Модальное окно: "Изменить только это событие" / "Все события серии" / "Это и все последующие"

**При удалении:**
- Модальное окно: "Удалить только это событие" / "Все события серии" / "Это и все последующие"

**В календаре:**
- Повторяющиеся события отображаются с иконкой 🔄
- При наведении — подсказка с правилом повторения ("Каждый Пн и Ср")

### 6.6 API-эндпоинты

| Метод | URL | Назначение |
|-------|-----|-----------|
| POST | `/crm/calendar-event` | Создание (с recurrenceRule) |
| PATCH | `/crm/calendar-event/byId/:id` | Обновление (mode: single/all/following) |
| DELETE | `/crm/calendar-event/byId/:id` | Удаление (mode: single/all/following) |
| GET | `/crm/calendar-event/range` | Возвращает в т.ч. виртуальные экземпляры |

Параметр `recurrenceMode` в PATCH/DELETE: `"single"` | `"all"` | `"following"`

## 7. Утренняя сводка дня в Telegram

### 7.1 Описание

Каждое утро (в 08:30 ТШК) каждый сотрудник получает в Telegram сводку своего дня: встречи, звонки, задачи с дедлайном сегодня, напоминания.

### 7.2 Формат сообщения

```
🌅 Доброе утро! Ваш план на сегодня, 4 февраля:

📅 Встречи (2):
  10:00 — 11:00  Планёрка отдела | 📍 Переговорная 1
  14:00 — 15:30  Обсуждение поставки | 🏢 ООО "Ромашка" | 📍 Офис клиента

📞 Звонки (1):
  11:30  Уточнить сроки отгрузки | 🏢 ООО "Василёк"

📋 Задачи с дедлайном сегодня (3):
  ⚡ [Срочный] Подготовить КП для ООО "Ромашка"
  📌 [Высокий] Согласовать условия поставки
  📎 [Обычный] Отправить счёт на оплату

🔔 Напоминания (1):
  16:00  Позвонить в банк по поводу гарантии

Всего: 2 встречи, 1 звонок, 3 задачи, 1 напоминание
```

Если план пустой — сообщение не отправляется.

### 7.3 Настройки пользователя

| Настройка | Тип | Default |
|-----------|-----|---------|
| Получать утреннюю сводку | Boolean | true |
| Время отправки | String (HH:mm) | "08:30" |

Хранение: новые поля в модели `User` или отдельная таблица `user_notification_settings`.

### 7.4 API

| Метод | URL | Назначение |
|-------|-----|-----------|
| GET | `/cron/morning-digest` | Крон: отправка утренних сводок |
| PATCH | `/user/notification-settings` | Обновить настройки уведомлений |
| GET | `/user/notification-settings` | Получить настройки |

### 7.5 Backend

Новый сервис: `apps/api/src/crm/services/morning-digest.service.ts`

- Метод `sendDigests()` — вызывается кроном
- Для каждого пользователя с включённой сводкой:
  - Запрос событий на сегодня (CalendarEventService.getTodayPlan)
  - Форматирование сообщения
  - Отправка через TelegramService
- Крон запускается в 08:00 МСК (или по настройке пользователя)

## 8. Аналитика и отчёты

### 8.1 Описание

Раздел аналитики для руководителей — статистика по задачам и событиям подчинённых за выбранный период.

### 8.2 Виджеты аналитики

**Статистика задач:**
- Количество задач по статусам (в работе / выполнено / просрочено / отменено)
- Средняя скорость выполнения (дни от создания до завершения)
- Процент задач выполненных в срок vs просроченных
- Топ сотрудников по выполненным задачам
- График: динамика создания/завершения задач по неделям

**Статистика событий:**
- Количество встреч / звонков / заметок за период
- Среднее количество встреч в день на сотрудника
- Количество отменённых vs проведённых встреч
- Активность по дням недели (в какие дни больше встреч)

### 8.3 Интерфейс

- Отдельная страница `/calendar/analytics` (доступ: руководитель + админ)
- Фильтры: период (неделя / месяц / квартал / произвольный), сотрудник / отдел
- Графики: Chart.js или Recharts
- Экспорт в Excel (опционально)

### 8.4 API-эндпоинты

| Метод | URL | Назначение |
|-------|-----|-----------|
| GET | `/crm/analytics/tasks?from=&to=&userId=` | Статистика задач |
| GET | `/crm/analytics/events?from=&to=&userId=` | Статистика событий |
| GET | `/crm/analytics/summary?from=&to=&userId=` | Общая сводка |

### 8.5 Права доступа

| Роль | Доступ |
|------|--------|
| Сотрудник | Только своя статистика |
| Руководитель | Своя + подчинённых |
| Админ | Все сотрудники + по отделам |

## 9. Drag & Drop в календаре

### 9.1 Описание

Перетаскивание событий в календаре для быстрого изменения даты/времени.

### 9.2 Функциональность

- В виде "Неделя" и "День": перетаскивание события на другое время/дату
- В виде "Месяц": перетаскивание на другой день
- При отпускании — автоматический `PATCH` с новыми `dateStart` / `dateEnd`
- Визуальная обратная связь: тень, подсветка слота
- Для задач (read-only в Фазе 1) — перетаскивание обновляет `deadline`
- Только свои события или события подчинённых (для руководителей)

### 9.3 Технические требования

- Использовать `react-dnd` или нативный HTML5 Drag & Drop
- Оптимистичное обновление UI (до ответа сервера)
- Откат при ошибке API
